# 分层设计概念

## 分层设计方法论

### Channel-based 方法论
- 成熟稳定的方法论
- 易于集成硬化 IP
- 通常不需要 Feedthrough 缓冲
- 缺点：面积和时序结果可能不够优化

### Channel-less 方法论
- 可节省 5-7% 的芯片面积
- 无顶层逻辑
- 要求：
  - 多扇出线网跨分区边界的精心规划
  - 精确的引脚对齐（用于相邻分区）
  - 复杂的主-克隆（重复块）配置
  - 高度定制的时钟树
  - 迭代设计流程
  - 硬化 IP 的引脚连接
- 需要 Verilog 模块包装器用于 LEC 验证

### Narrow Channel 方法论
- Channel-based 和 Channel-less 的平衡方案
- 优点：
  - 对引脚位置和 Feedthrough 路径无严格要求
  - 易于集成硬化 IP
  - 灵活的时钟树拓扑选择
  - 快速收敛
- 缺点：相比 Channel-less 设计有轻微面积损失

## 分层分区流程和能力

### 分层分区

**核心功能**：
- 将设计逻辑和物理分区为顶层设计和各分区块
- 将所有相关设计数据推送到分区块
- 支持：
  - 分区块和黑盒定义
  - 多种方向和非矩形形状
  - 多个实例化分区
  - 多层级分区
  - 分区保护带（Guard-bands）
- 为顶层和各块创建独立目录，支持不同设计团队并行实现

### 引脚分配

**自动引脚分配能力**：
- 相邻（无通道）设计
- 多层级分层设计
- 主设计和克隆设计
- 翻转芯片（面积 IO 设计）
- 非矩形分区形状
- 基于布线的引脚分配
- 基于布局的引脚分配

**约束管理**：
- 支持多种强大约束指导引脚分配优化
- 约束可在设计、单个分区或特定引脚/引脚组级别设置
- 冲突通过明确的优先级规则处理

**交互式引脚编辑器**：
- 支持自定义引脚放置
- 在指定区域跨层分散引脚组
- 放置非首选层的引脚
- 遵守用户指定的引脚约束

**引脚检查和合法化**：
- 检查已分配引脚
- 合法化违反约束的引脚
- 跨布线通道对齐引脚

### Feedthrough 插入

**insertPtnFeedthrough 命令**：
- 在分区中插入 Feedthrough 线网和缓冲
- 避免线网在分区区域上布线（避免与分区布线资源冲突）
- 避免绕过分区（避免更长的布线和延迟）
- 在无通道设计中必需，在基于通道的设计中也可能有用

**高级能力**：
- 基于布局或布线插入 Feedthrough
- 多电源电压设计的正确缓冲插入
- 跨相邻多实例化（主/克隆）分区优化缓冲复用
- 多扇出线网的缓冲复用

### 时序预算分配

- 在分区设计时自动将芯片级时序约束映射到块级约束
- 使用基于路径的方法分配预算
- 在早期非优化设计上执行虚拟设计优化，获得更准确的分区时序预算
- 支持更快的模式（避免虚拟优化）
- 提高设计收敛性

### 分区组装

**assembleDesign 能力**：
- 为嵌套分区恢复分区数据
- 流程：恢复顶层设计 → 组装父分区 → 恢复所有子节点分区
- 确保主设计和克隆设计（可能在不同分区的不同层级）正确组装

## 分层模型和接口

### Interface Logic Model (ILM)

**定义**：
- 块的结构表示
- 包含 I/O 时序路径上的实例、时钟树实例、影响 I/O 时序路径信号完整性的实例或线网
- 块时序特性的紧凑准确表示

**优势**：
- 比黑盒流程更准确的分析
- 比组合 .lib 或 .cdb 方法更了解 SI
- 可建模块内时钟生成器
- 更准确的时序和 SI 减少设计迭代次数
- 无需块表征
- 基于实际设计数据
- 可用于大型设计的初期原型设计阶段
- 仅允许修改顶层数据
- 完全保留已实现的分区
- 使用原始约束文件进行顶层分析
- 无时序异常抽象

**局限**：
- 无法修改以在顶层关闭时序
- 可能导致块和顶层优化之间的迭代

### Flexible Interface Logic Model (FlexILM)

**定义**：
- 减少的网表，保留接口路径上的逻辑，移除内部路径上的逻辑
- 移除的实例被放置阻挡替代，避免与新优化逻辑冲突
- 移除的线网布线被 RC 网格替代，改进 RC 提取相关性

**优势**：
- 单遍分层解决方案
- 减少时序图和物理数据中的内存
- 在顶层设计中优化接口路径
- 网表和放置变化可自动 ECO 回分区块

### FlexModel

**定义**：
- 可用于分层分区实现流程或原型设计流程
- 网表可减少至 20 倍
- 包含：宏、接口标准单元、FlexFillers

**FlexFillers**：
- 填充内部寄存器到寄存器逻辑的空间
- 无关联的时序模型
- 布局器将其紧密放置在一个组中
- 帮助准确的时序和面积估计

**性能**：
- FlexModel 网表通常是完整网表实例数的十分之一
- 用于早期设计规划，减少运行时间和内存
- 支持准确的时序、面积和拥塞分析
- 结果的可实现性强，通常一次收敛

**创建时机**：
- 可在设计早期创建，即使网表不完整

**原型设计流程阶段**：
1. 创建 FlexModels
   - 识别模型
   - 创建模型
   - 创建时序线网延迟模型（psPM）
2. 调试约束和规划设计
   - 调试约束
   - 生成初始 floorplan
3. 分析和调整 Floorplan
   - 手动移动 FlexModels 缩短时序路径
   - 重新分析 floorplan
4. 定义分区
   - 基于 FlexModel 放置定义分区
   - 可选：在分区围栏内重新放置宏
   - 可选：在分区边界内手动重新调整 FlexModel 区域
   - 生成分区围栏
5. 完成和保存分区
   - 放置和早期全局布线
   - 提交/保存分区
   - 引脚分配、Feedthrough 插入、预算分配
   - 电源、总线和流水线规划

### Pico-second Per Micron (psPM) 模型

**定义**：
- 用于快速时序估计的时序线网延迟模型
- 无需用户优化设计
- 可用于分层实现流程和/或原型设计流程

**特征**：
- 建模虚拟缓冲效应（主导长线网）
- 基于线网长度、布线层和扇出数量表征
- 使用当前工艺库创建 2 引脚线网和多扇出线网的样本设计
- 使用 GigaOpt 优化这些线网以获得 psPM 值

**计算公式**：
- psPM = (总延迟包括虚拟缓冲延迟) / (线网长度)
- 考虑输入转换
- 使用 optDesign -preCTS 命令优化延迟和转换

**准确性**：
- 与 optDesign -preCTS 结果相关性约 10%
- 支持快速 TAT 的准确时序估计

**创建方式**：
- 使用 create_ps_per_micron_model 命令
- 在基于分区的 FlexModel 生成步骤中自动生成

## 分层数据库和芯片规划

### Stylus Hierarchical Database (SHDB) 流程

**概述**：
- 轻松导航分层数据库
- 选择性打开特定块的特定块模型以节省内存
- 轻松遍历多层级分层
- 自动同步顶层到块级或反向的数据库变化

**块级操作**：
- 在独立会话中打开块级数据库
- 修改数据库（floorplan、引脚位置、电源布线、place_opt_design）
- 保存变化并创建块模型
- 同步变化回顶层设计
- 生成 ECO 差异报告

**功能特性**：
- 提供分层的鸟瞰图
- 支持跨边界和多层级逻辑分层的 DRC/时序调试
- 支持多种块级模型（ILM、FlexILM、LEF、etm、pnr DB、自定义模型）
- 在顶层设计配置块模型，支持混合模型类型和选择性模型视图/标签
- 灵活的分层数据库配置保存和恢复
- 全局块数据库存储库管理和版本控制（SVN）
- 块数据库存储库打包和库重新链接工具用于可移植性

**许可证要求**：
- 基础 Innovus Digital Implementation System 许可证
- 使用 commit_module_model 命令处理 pnr 和 FlexILM 模型类型需要额外的分层许可证

### 芯片规划

**目标**：
- 生产性芯片规划
- 并发处理多个设计目标（容量/周转时间/抽象模型/优化）

**支持的流程方法论**：
- FlexModel
- psPM 模型
- 原型设计流程

### 芯片组装

**定义**：
- 顶层和所有块的详细信息汇总
- 用于完整芯片提取、电源、时序和串扰分析

**assembleDesign 命令**：
- 将分区 Verilog 网表文件连接回顶层
- 将设计数据与原始顶层设计合并
- 默认使用 DEF 文件数据，可指定使用 Innovus 数据或 OpenAccess 数据库格式
- 切割顶层行，恢复块级设计行
- 保留分区块级设计的扫描链信息
- 调整扫描链起点和终点回连接到扫描链点的实例

**DEF 合并能力**：
- 无需分区 LEF 文件
- 降低峰值内存需求
- 快速周转时间

**前置条件**：
- 保存顶层和块级设计（saveDesign -def）
- 从包含完整芯片级 floorplan 的目录运行 assembleDesign
- 使用 Tempus ECO 流程时，assembleDesign 后运行 checkPlace 并修复所有违反

### SHDB 存储库管理

**命令**：
- set_module_model -default_dir：指定模块模型加载/保存的默认位置
- savePartition –module_model_tag：保存顶层和块级 pnr 模块模型到全局分层数据库存储库
- create_module_model：创建并保存指定模块模型到全局数据库
- restore_module_model：以一致方式恢复顶层（块 db）数据库

**存储库结构**：
- 默认目录为顶层
- 所有单元或模块模型在默认目录下配置
- 每个单元名称关联用户定义的标签名称（简单版本控制）
- 标签名称可为设计阶段/状态名称（如 'prects'、'cts' 等）
- 'latest' 标签为保留字，指向最新保存的标签目录

**版本控制**：
- 支持 Apache Subversion (SVN)
- 中央存储库用于检出数据到本地工作目录
- 块级设计生成模型后检入更新的块数据模型
- 按顺序检入/检出以避免数据锁定问题
